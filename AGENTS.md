## 编码规范

### 1. 先思考，再动手

实现前先明确：
- 主动说明假设。不确定时提问，而非猜测。
- 若存在多种理解方式，列出来让用户选，不要默默挑一个。
- 若有更简单的方案，说出来，必要时推回去。
- 遇到模糊的地方，停下来，点名哪里不清楚，再问清楚。

### 2. 简洁优先

最少的代码解决问题，不写任何推测性内容：
- 不做需求之外的功能。
- 单次使用的代码不做抽象。
- 不加没有被要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 写了 200 行却可以用 50 行解决？重写。

### 3. 精准修改

只改必须改的，只清理自己制造的混乱：
- 不"顺手优化"相邻代码、注释或格式。
- 不重构没有问题的东西。
- 保持现有风格，即使你有不同偏好。
- 发现无关的死代码？提出来，不要自行删除。

自己的改动产生的孤儿代码（未使用的 import、变量、函数）需要清理，但原本存在的死代码除非被要求否则不动。

判断标准：每一行改动都应该能直接追溯到用户的需求。

### 4. 前端展示规范

- **时间显示**：前端页面上的所有时间显示都必须统一处理为东八区北京时间。带 `Z` 或显式 offset 的时间先换算到 `Asia/Shanghai` 后展示；无时区的后端时间字符串按北京时间墙钟时间展示，避免浏览器本地时区造成偏移。
- **数值显示**：各页面的数值必须使用统一过滤/格式化处理，默认最多保留 2 位小数；展示时不补零。若业务未另行明确要求，不得直接输出后端原始长小数。

### 5. 文档同步规范

**每次代码修改完成后，必须同步更新 `docs/部署文档.md` 和 `README.md`**：

- 新增或变更接口（路径、参数、响应、错误码）→ 更新 API 文档
- 新增或变更用户可见功能（页面、按钮、行为、状态）→ 更新用户手册
- 纯内部实现优化（性能、超时保护、重构等）→ 若影响用户可观察行为（如错误处理、超时表现），需更新用户手册相关说明

文档更新与功能代码提交在同一个 commit 中完成。

### 6. 目标驱动执行

把任务转化为可验证的目标，执行直到通过验证：
- "添加校验" → "为非法输入写测试，然后让测试通过"
- "修复 Bug" → "写一个能复现问题的测试，然后让测试通过"
- "重构 X" → "确认重构前后测试均通过"

多步骤任务要先简述计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确的成功标准让你能独立循环推进；模糊的标准（"让它能跑"）会导致反复确认。

## 文件处理边界

- 执行所有项目任务时，项目内文件可由代理根据任务目标自主决定读取、修改、创建、移动或删除。
- 项目外文件只能读取，不能修改、移动或删除。
- 只有当用户给出直接、明确的命令要求处理某个项目外文件时，才允许按该命令处理对应文件。
- 若项目外文件处理请求含糊、范围过大或存在破坏性风险，必须先向用户确认，不能自行执行。
