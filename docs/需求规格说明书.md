# iBreeze 需求规格说明书

> 本文档面向**无代码经验的读者**，目标是让读者能够依据本文与指定技术栈，**从零重建整个 iBreeze 项目**，并且文档内容与代码逐一对齐。
> 文档中的所有接口、表结构、状态机、业务规则均来自 `sidecar/`、`apps/desktop/` 与 `docs/` 下的实际实现，可据此逐条核对。

---

## 第 1 章 概述与目标

### 1.1 产品定位

iBreeze 是一个**多租户 AI 协作后端平台**。它把"组织 / 权限 / 能力 / 后端 Provider / 会话 / 任务 / 知识库 / 治理"统一编排在一个可嵌入桌面端的 sidecar 进程中，对外暴露一套统一的 JSON-RPC 接口。

- **后端进程（sidecar）**：`sidecar/` —— 一个常驻的 async Python 服务，监听 Unix Domain Socket（UDS）。
- **桌面前端（Tauri + React）**：`apps/desktop/` —— Tauri 2 外壳内嵌 React 19 单页应用，通过 Tauri `invoke` 调用 Rust 桥接，再由 Rust 转发到 sidecar 的 UDS。

### 1.2 技术栈

| 层 | 技术 |
|----|------|
| 后端语言 | Python 3.11+，全程 async（asyncio） |
| 关系存储 | SQLite（`aiosqlite` 异步驱动），运行在 `/tmp/acos.db` |
| 向量 / 文档存储 | LanceDB（sidecar 内嵌，用于知识库、文档、citation 索引） |
| 进程间通信 | Unix Domain Socket（`/tmp/acos.sock`）+ JSON-RPC 2.0 风格自研协议 |
| 桌面外壳 | Tauri 2（Rust） |
| 前端框架 | React 19 |
| 前端状态 | Zustand（全局 store）+ TanStack Query（服务端缓存/请求） |
| 包管理 / 构建 | npm（前端）、cargo（Rust）、tauri CLI |

### 1.3 应用标识

- `app` 标识：`com.ibreeze.desktop`
- `productName`：`iBreeze`

### 1.4 设计目标

1. **强一致多租户**：所有数据归属 `company_id`，跨公司访问一律拒绝。
2. **可审计**：关键写操作进入审计表与领域事件表。
3. **可幂等**：所有"写方法"必须携带 `idempotency_key`，服务端统一预约并回填，崩溃重试不重复生效。
4. **可治理**：预算、审批、权限引擎统一收敛到治理层。
5. **可重建**：所有表结构由 `sidecar/migrations/` 顺序迁移生成，文档第 4 章与迁移文件一一对应。

### 1.5 关键非功能性约定

- **前端时间显示**：所有时间统一换算到**东八区（Asia/Shanghai）**展示，禁止浏览器本地时区偏移。带 `Z` 或 offset 的先换算，无时区的后端字符串按北京时间墙钟展示。
- **前端数值显示**：所有数值经统一过滤器处理，**最多保留 2 位小数且不补零**。
- **错误码**：RPC 失败统一通过 `error` 字段返回，含 `code` 与 `message`（详见 3.4 / 各域契约）。

### 2.2 绑定链与归属

- **Provider ← Template ← Employee**：员工使用模板，模板绑定 Provider；三者全部归属某个 `Company`。
- 所有跨域查询必须以 `company_id` 为首要过滤条件。
- **软删除**：`company`、`employee` 等核心实体采用 `deleted_at` 软删除，物理删除仅在解散（dissolve）收尾阶段或显式 archive 流程中触发。

### 2.3 进程边界（谁调谁）

```
React 19 (Zustand + TanStack Query)
   │  invoke('sys_rpc_call', { method, params: JSON.stringify(params) })
   ▼
Tauri Rust 桥 (apps/desktop/src-tauri)
   │  UDS 写 /tmp/acos.sock
   ▼
sidecar RPCServer (async Python)
   ├─ 幂等预约 / 校验 / 分发 (_dispatch)
   ├─ 各 Methods 集合（org/cap/backend/provider/session/task/workflow/...）
   ├─ 治理层 (gov): 预算 / 审批 / 权限引擎
   ├─ SQLite (/tmp/acos.db)  ── aiosqlite
   └─ LanceDB                 ── 向量/文档
```

### 2.4 接口调用约定（通用）

所有前端调用统一走：

```ts
invoke('sys_rpc_call', { method: 'org.company.list', params: JSON.stringify({ ... }) })
```

- `method`：字符串，形如 `org.company.list`（见第 3 章完整清单，共 **175** 个方法）。
- `params`：**JSON 字符串**（不是对象），服务端反序列化。
- 写方法必须额外携带 `idempotency_key`（字符串）。服务端在 `_dispatch` 中：
  1. 若方法是写方法且带 `idempotency_key` → 先预约幂等记录（保留成功/失败结果）；
  2. 执行 handler；
  3. 通过 `complete_idempotency(...)` 回填结果。
  4. 同一 `idempotency_key` 重复提交 → 直接返回首次结果，不二次执行。

## 第 5 章 状态机总览

> 下列状态转移常量来自对应 `models.py` / `service.py`，是后端强制约束。

### 5.1 company

`initializing → active → dissolving → dissolved`，外加软删 `deleted`（通过 `deleted_at` 标记）。`dissolve` 触发 `dissolution_watermarks` 6 个消费者流水线，完成 `employee_drain` 后落 `dissolved`。

### 5.2 department

`active ↔ frozen`，`active/frozen → archived`。冻结态禁止在其下创建子部门。

### 5.3 employee

`created → active → suspended → archived → deleted(软)`；`suspended → active(resume)`。`deleted` 为 `deleted_at` 软删。

### 5.4 task / task_node / workflow run

- **TASK_TRANSITIONS**：含 `cancelling` 等；`create → start → complete`，`→ cancelling → cancelled`，失败可 `retrySubtask`。
- **NODE_TRANSITIONS**：子任务节点状态转移（与 task 联动）。
- **RUN_TRANSITIONS**：工作流运行态。
- **GENERATION_TRANSITIONS**：plan / 生成态（见 `task/models.py`）。

### 5.5 backend

`enabled → draining → disabled → archived`；`draining` 期内只收尾不接新。

### 5.6 capability / skill / prompt

统一：`draft → review → published → deprecated → archived`；`saveDraft` 留在 `draft`；`submitReview` 进 `review`；`publish` 进 `published`。

### 5.7 session

`active → running → waiting_backend / waiting_approval → dormant → archived`，异常 `failed → recovering → active`。当员工 `suspended` 或模板 `archived` 时会话置 `READONLY`。

### 5.8 grant

`active → expired / revoked`（`revoked_at` 标记撤销）。

### 5.9 company dissolution 流水线

`dissolving` 经 `dissolution_watermarks` 6 个消费者顺序处理（含 `employee_drain`），全部完成后转 `dissolved`。

## 第 6 章 端到端业务规则

> 以下不变量摘自 `docs/功能链路测试用例.md`，是系统强制行为，重建时必须保证。

1. **跨公司拒绝**：任何以 `company_id` 过滤的查询/写操作，跨公司访问一律拒绝（返回权限错误）。
2. **预算拦截**：任务创建/预算预留超出 `gov.budgetPolicy` 时，触发 `gov.approval` 审批或拒绝（预算修订需 `budget_revision_lock`）。
3. **冻结部门禁建子部门**：父部门 `frozen` 时 `org.department.create` 拒绝。
4. **含员工部门禁删**：部门下仍有员工或子部门时 `org.department.delete` 拒绝。
5. **会话 READONLY 联动**：会话所属员工 `suspended` 或所用模板 `archived` → 会话进入只读，禁止 `sendMessage` 改动。
6. **deadletter 联动 task**：工作流节点失败进入 `dead_letters`，经 `workflow.deadletter.resolve` 处理后回写 task 状态。
7. **dissolve → employee_drain**：`org.company.dissolve` 触发 `dissolution_watermarks`，先完成员工排水再 `dissolved`，期间禁止新写。
8. **权限引擎统一解析**：`org.permission.resolve` 委托 `cap.engine.resolve`，所有能力/资源访问经同一引擎裁决。
9. **幂等回填**：写方法带 `idempotency_key` 重复提交，返回首次结果不重复生效（见 1.5 / 3.4 约定）。
10. **审批联动**：plan 校验（`workflow.plan.validate`）、预算修订等需要人工决策时创建 `approval`，`approval.resolve` 决议后才继续。

## 第 7 章 构建与部署

### 7.1 后端 sidecar 运行

- SQLite 数据库：`/tmp/acos.db`（aiosqlite 异步访问）。
- LanceDB：sidecar 内嵌，用于知识库/文档/向量索引。
- UDS 监听：`/tmp/acos.sock`。
- 启动顺序：应用（Tauri/Rust）拉起 sidecar 进程 → sidecar 执行 `migrations` 全部迁移 → 监听 UDS → 注册全部 175 方法 → 健康检查 `sys.health` 通过。

### 7.2 迁移执行

- 所有表结构变更**只**通过 `sidecar/migrations/YYYYMMDDHHMMSS_description.sql`。
- 命名格式 `YYYYMMDDHHMMSS_description.sql`（现有为 `0001..0062` 顺序编号）。
- 脚本幂等：`CREATE TABLE IF NOT EXISTS`、`CREATE INDEX IF NOT EXISTS`；`ADD COLUMN` 先 `PRAGMA table_info` 判断；`DROP` 旧表放最后且确认新表就绪后。
- 迁移失败修复脚本后可直接重跑。

### 7.3 桌面端构建（Tauri 2）

```bash
# 在 apps/desktop 下
npm install
npm run tauri build
```

产物位于：

```
apps/desktop/src-tauri/target/release/bundle/
```

- `app` 标识：`com.ibreeze.desktop`
- `productName`：`iBreeze`

### 7.4 前端调用约定回顾

- 所有 RPC 经 `invoke('sys_rpc_call', { method, params: JSON.stringify(params) })`。
- 写方法必须带 `idempotency_key`；前端时间统一东八区；数值最多 2 位小数不补零。

### 7.5 关闭与运维

- 优雅关闭：上层调用 `sys.shutdown` → sidecar 进入关闭流程（见 `supervisor.py`：`sys.shutdown → SIGTERM → SIGKILL`）。
- 健康探测：上层轮询 `sys.health` 判断是否存活。
