# iBreeze 功能链路测试用例文档

> 本文档基于《AI 公司桌面应用设计方案》《AI 公司桌面应用-实施计划》以及当前代码实现（sidecar 后端 + Tauri/React 前端 + Admin Backend + Admin Frontend），为 **产品预期最终形态** 编写的端到端功能链路测试用例。
>
> 每一组用例从"用户视角的完整操作链路"出发，串起多个 RPC / 页面，验证跨实体的业务闭环。用例可执行性参考了现有测试基线：**Sidecar 849 passed / Admin Backend 157 passed / Desktop 193 passed / Integration 49 passed**。
>
> 约定：
> - **链路用例（E2E-xx）**：一条跨多页面/多 RPC 的完整业务链路，是本文档重点。
> - **场景用例（SC-xx）**：针对单条链路中的关键分支/异常/边界。
> - **契约用例（Contract-xx）**：端到端跨系统边界的契约验证，确保各系统间数据归属、同步、接口映射的正确性。
> - 前置：`org.company.create` 创建公司（status=initializing），`org.company.activate` 激活（建根部门 + 首任负责人员工）是绝大多数链路的前置条件。
> - 时间显示遵循东八区规范，数值最多 2 位小数不补零（前端统一处理）。

## 一、组织与身份链路

### E2E-01 公司从创建到激活的完整生命周期

**目标**：验证公司注册→激活→部门/负责人建立的闭环。

| 步骤 | 操作（页面/组件） | RPC | 预期验证 |
|---|---|---|---|
| 1 | 公司列表页点「新建公司」，填名称 | `org.company.create` | 返回 `company_id`，`status=initializing`，`version=1` |
| 2 | 公司详情点「激活」 | `org.company.activate`（带 `leader.name`） | `status=active`，返回 `leader_employee_id` |
| 3 | 查根部门 | `org.department.list` | 存在 `root_department_id`，`leader_employee_id` 已写入 |
| 4 | 查员工 | `org.employee.list` | 出现 1 名 `employee_type=company_leader`、`status=active` 的负责人 |
| 5 | 公司列表聚合 | `org.company.list` | Dashboard 公司数 +1 |

**SC-01-1（激活幂等）**：对 `initializing` 公司已激活后再激活 → 应报错或幂等成功，不产生第 2 个负责人。
**SC-01-2（版本冲突）**：激活时传错误 `expected_version` → CAS 失败，返回乐观锁冲突错误。
**SC-01-3（无 leader 激活）**：不传 `leader.name` → 使用默认名 "owner"，负责人仍创建成功。

### E2E-02 部门树创建与闭包关系维护

**目标**：验证部门层级、闭包表、负责人、环检测的闭环。

| 步骤 | 操作 | RPC | 预期验证 |
|---|---|---|---|
| 1 | 在根部门下建「研发部」 | `org.department.create`（parent=root） | 部门创建成功 |
| 2 | 在研发部下建「前端组」 | `org.department.create`（parent=研发部） | 成功 |
| 3 | 查部门列表 | `org.department.list` | 闭包表 `department_closure` 含 (root→研发部, root→前端组, 研发部→前端组) |
| 4 | 设部门 leader | `org.department.setLeader` | `leader_employee_id` 写入 |
| 5 | 移动部门 | `org.department.move` | 闭包表 depth 正确重算 |
| 6 | **环检测** | `org.department.move`（把 root 移到 前端组 下） | 拒绝，返回环错误，不破坏现有闭包 |

**SC-02-1（冻结/解冻）**：`org.department.freeze` → 子部门不可新建；`unfreeze` 恢复。
**SC-02-2（删除）**：删除含员工的部门 → 拒绝或级联处理；空部门删除成功。

### E2E-03 员工创建 → 绑定基座模板 → 能力快照注入

**目标**：验证 `Provider ← Template ← Employee` 绑定链在员工创建时落地。

| 步骤 | 操作 | RPC | 预期验证 |
|---|---|---|---|
| 1 | 先建 Provider（见 E2E-10）、建 Capability（见 E2E-20）、建 Template 绑二者 | `org.template.create` | 模板 `provider_id`/`capability_id`/`model` 写入 |
| 2 | 员工页点「新建员工」，选部门 + 选基座模板 | `org.employee.create`（带 `template_id`） | 员工创建成功，`template_id` 落库 |
| 3 | 查员工记录 | 直接查 `employees` 表 | `capability_snapshot` 已注入模板的 capability 快照（非空 JSON） |
| 4 | 员工详情页 | `org.employee.get` | 展示基座模板名、部门、能力快照 |

**SC-03-1（无模板创建）**：`template_id` 为空 → 员工仍可创建，`capability_snapshot` 为 `{}`。
**SC-03-2（跨公司模板）**：用他司模板 id 建员工 → 拒绝或快照为空，不得串公司。
**SC-03-3（汇报链环）**：`org.employee.setManager` 制造 A→B→A 循环 → 拒绝。
**SC-03-4（状态机）**：员工 `created → activate → suspend → resume → archive → delete(软删)` 各状态迁移正确，`deleted_at` 软删后 `employee.list` 不返回。

## 二、供应商（Provider）与后端（Backend）链路

### E2E-10 API Key 形式 Provider 配置与模型实时查询

**目标**：验证 API 形式 Provider 的"选供应商→填 key→实时查模型→保存"闭环，覆盖你刚确认的需求。

| 步骤 | 操作（Provider 页） | RPC | 预期验证 |
|---|---|---|---|
| 1 | 点「新建 Provider」，类型选「API Key 形式」 | — | 显示供应商下拉 |
| 2 | 选供应商 = DeepSeek，填 API Key | — | key 仅存 Keychain，不落库 |
| 3 | 点「查询可用模型」 | `provider.models.fetch`（api_vendor=deepseek, api_key） | 调用 `https://api.deepseek.com/models`，返回模型下拉 |
| 4 | 从下拉选一个模型 | — | `modelValue` 选中 |
| 5 | 确认创建 | `provider.create`（config 含 `api_vendor`+`base_url`）+ `provider.credential.set` | Provider 落库 `provider_type=api`，`config.api_vendor=deepseek`；凭证写入 Keychain |
| 6 | 查模型列表 | `provider.model.list` | 该公司可见模型含所选模型 |

**SC-10-1（OpenAI 实时查）**：供应商=OpenAI → 请求 `https://api.openai.com/v1/models`，Bearer 鉴权。
**SC-10-2（Anthropic 实时查）**：供应商=Anthropic → 请求 `https://api.anthropic.com/v1/models`，`x-api-key`+`anthropic-version` 头，返回 `id`+`display_name`。
**SC-10-3（第三方兼容）**：供应商=第三方，手动填 Base URL（OpenAI 兼容）→ 请求 `{base_url}/models`。
**SC-10-4（查询失败降级）**：填错误 key → `provider.models.fetch` 返回 `source=fallback` + 内置清单，页面提示"查询失败，已用内置清单"，仍可手动填模型名。
**SC-10-5（无 key）**：不填 key 直接查询 → 直接返回内置兜底清单。
**SC-10-6（凭证安全）**：`provider.credential.get` 能取回；`provider.credential.revoke` 后取回为 null；`provider.create` 落库的 `config` 中不含明文 key。

### E2E-11 CLI（Agent）形式 Provider 配置

**目标**：验证调用本机 agent 工具的 Provider 创建闭环。

| 步骤 | 操作 | RPC | 预期验证 |
|---|---|---|---|
| 1 | 点「新建 Provider」，类型选「调用 Agent 形式」 | — | 显示 agent 下拉 |
| 2 | 点「查询可用模型」/加载 agent 列表 | `provider.agent.list` | 返回 4 个固定 agent（cursor-cli/claude-code/codex-cli/opencode）及其模型 |
| 3 | 选 agent = opencode，选模型 = anthropic/claude-sonnet-5 | — | 模型下拉含内置清单 |

**SC-11-1（未知 agent 拒绝）**：`provider.create` 传 `agent=xxx` → 校验失败。
**SC-11-2（cli 缺 model）**：不传 model → 校验失败。
**SC-11-3（自定义模型）**：cli 模型选「自定义」→ 手动输入模型名生效。

### E2E-12 Backend 生命周期：创建→探测→启用→设为默认→排空→归档

**目标**：验证 backend 与 provider 关联及状态机。

| 步骤 | 操作（Provider 页） | RPC | 预期验证 |
|---|---|---|---|
| 1 | 点「新建 Backend」，选类型 local_process，关联 provider | `backend.create` | 创建成功，`company_id` 归属当前公司 |
| 2 | 点「探测」 | `backend.probe` | 返回 available/healthy |
| 3 | 点「启用」 | `backend.enable` | `status=enabled` |
| 4 | 设为默认 | `backend.setDefault` | `company_backend_defaults` 唯一约束，旧默认被替换 |
| 5 | 点「排空」 | `backend.drain` | `status=draining`，不再接新租赁 |
| 6 | 点「归档」 | `backend.archive` | `status=archived`；有 held lease 时拒绝归档 |

**SC-12-1（健康门禁）**：`enable` 前若探测非 healthy → 拒绝。
**SC-12-2（默认唯一）**：并发设默认 → version CAS 保证唯一。
**SC-12-3（可用性检查）**：`backend.checkAvailability` 返回当前容量/健康。

## 六、会话（Session）链路

### E2E-40 会话创建 → 发消息 → 安全上下文重算 → 转录

**目标**：验证会话完整交互闭环（session.*）。

| 步骤 | 操作（会话页） | RPC | 预期验证 |
|---|---|---|---|
| 1 | 选员工 + 选已有会话/新建线程 | `session.list` / `session.get` | 线程列表/详情 |
| 2 | 发送消息 | `session.sendMessage`（employee_id, content） | 服务端 `_resolve_security_context` 按 employee/capability/provider/policy/有效授权重算 key（**不接受客户端 key**）；占用 backend lease；经 FakeProvider 返回 |
| 3 | 取转录 | `session.transcript.get` | 含用户消息 + 助手回复 + security_context |
| 4 | 取消 | `session.cancel` | 会话终止 |
| 5 | 恢复 | `session.resume` | 重新激活 dormant 会话 |

**SC-40-1（单 active turn CAS）**：同一会话并发发两条消息 → 第二条因单 active turn 冲突被拒。
**SC-40-2（STALE/READONLY 分支）**：员工 suspended/模板 archived → sendMessage 返回只读/过期错误。
**SC-40-3（安全上下文服务端权威）**：客户端伪造 `security_context_key` → 被忽略，以服务端重算为准。
**SC-40-4（后端不可用）**：无 healthy backend → sendMessage 失败并触发 `human_intervention`（backend_recovery 子类）。

## 十三、端到端契约验证

> 以下契约用例验证跨系统边界的数据归属、同步、接口映射正确性，确保 Desktop（Sidecar）、Admin Backend、Admin Frontend 三端行为符合设计规范。

### Contract-01: Admin Backend → Sidecar 数据归属契约

**目标**：验证业务数据（Company/Department/Employee）由 Sidecar 创建和管理，Admin Backend 仅为只读镜像。

| 步骤 | 操作 | 验证点 | 预期结果 |
|---|---|---|---|
| 1 | 通过 Sidecar RPC `org.company.create` 创建公司 | Sidecar DB 写入 company | company 记录存在于 Sidecar DB |
| 2 | 调用 Admin Backend `GET /api/companies` | 读取 Admin Backend | 返回列表中包含步骤 1 创建的公司 |
| 3 | 调用 Admin Backend `POST /api/companies`（尝试创建） | 写入被拒绝 | 返回 405 或 400，Admin Backend 不具备 company 创建能力 |
| 4 | 调用 Admin Backend `DELETE /api/companies/{id}`（尝试删除） | 写入被拒绝 | 返回 405 或 404，Admin Backend 不具备 company 删除能力 |
| 5 | 验证 Department/Employee 同理 | `GET /api/departments` / `GET /api/employees` | 仅返回 Sidecar 创建的数据，POST/DELETE 均被拒绝 |

**判定标准**：Admin Backend 对 Company/Department/Employee 仅有 GET 权限；所有写操作由 Sidecar 接管。
