# iBreeze 项目测试用例

文档状态：正式测试基线

依据：

- `docs/设计方案/AI公司桌面应用设计方案.md`
- `docs/设计方案/AI公司桌面应用-实施计划.md`

本文档定义 iBreeze v1 的功能链路、多服务契约、安全恢复和交付门禁测试。测试实现不得通过降低预期、跳过失败或以 Mock 代替必须验证的真实边界来获得通过。

## 1. 测试范围与判定规则

### 1.1 被测系统

| 组件 | 职责 | 主要测试层 |
|---|---|---|
| Desktop WebView | 登录、公司、会话、任务、Runtime、Review、Workspace、知识与设置 UI | 单元、组件、E2E |
| Rust Desktop Core | Tauri Command、Profile、Keychain、Backend Client、Sidecar Supervisor、文件授权、Egress/Credential Broker | Rust 单元、契约、真机安全 |
| Python Sidecar | 本地领域、编排、Runtime Gateway、SQLite、Artifact、Review、知识与备份 | Python 单元、契约、集成、故障注入 |
| Backend API | 应用认证、管理员认证、用户、Catalog、Skill、Release、审计与健康检查 | Python 单元、PostgreSQL/MinIO 集成、OpenAPI |
| Admin Web | 管理员登录、用户管理、Catalog、Skill、兼容规则、发布、紧急禁用与审计 | 单元、组件、E2E |
| PostgreSQL/MinIO | 中心数据与发布对象 | 迁移、事务、灾备 |
| Codex/Claude Code/OpenCode/API Provider | 职员执行底座 | Fake 契约、受控真实 Adapter、发布冒烟 |

### 1.2 通过标准

1. 每个用例必须具有可重复的准备步骤、输入、预期结果和自动化证据。
2. 错误必须按设计方案第 20 章返回稳定错误码；不能以错误消息文本作为程序判断依据。
3. 跨公司、跨 Profile 和越权对象统一不得泄露目标是否存在。
4. 事务型用例必须同时验证数据库事实、DomainEvent、Outbox、投影和幂等记录的一致性。
5. 多服务契约必须同时验证请求 Schema、响应 Schema、认证、错误映射、超时、重试和幂等行为。
6. 必须使用真实 PostgreSQL 16、MinIO、SQLite 固定运行时和打包 Sidecar 完成对应集成门禁。
7. Fake Agent/Provider 用于确定性契约测试；设计指定的真实 CLI、Seatbelt、签名、公证和恢复门禁不能由 Fake 替代。
8. 任一测试命令非零退出即失败；验证脚本不得捕获失败后继续返回成功。
